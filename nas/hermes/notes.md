# Hermes Agent (NAS)

TrueNAS Scale Apps, **Custom App** (not in catalog). Installed 2026-04-24
via `midclt`. Sits next to LiteLLM (`nas/litellm/`) on the same NAS —
Hermes hits LiteLLM at `http://192.168.1.65:4000` with the LiteLLM master
key as its OpenAI-compatible API key.

## Endpoints

- **Gateway (direct LAN):** `http://192.168.1.65:8642`
  - `/health` — liveness (exact path not yet verified in our install; see
    [Hermes docs](https://hermes-agent.nousresearch.com/docs/))
  - `/v1/...` — OpenAI-compatible API (same shape as LiteLLM)
- **Admin UI behind Authelia 2FA:** `https://hermes.mati-lab.online` →
  Caddy on Pi → `192.168.1.65:8642`

## State on disk

Everything the agent mutates lives under `/mnt/fast/databases/hermes-data/`.
This single dir contains the whole deployment — memories, skills, sessions,
gateway state, SQLite db, config, and secrets.

| Path | Content |
|---|---|
| `.env` | `OPENAI_BASE_URL=http://192.168.1.65:4000/v1`, `OPENAI_API_KEY=<LITELLM_MASTER_KEY>`, Telegram vars later. Never committed. |
| `config.yaml` | Generated by `hermes setup`. ~50 KB of model routing, delegation, auxiliary, platforms config. Managed by Hermes, not us. |
| `state.db*` | SQLite — agent's memory + session state. **This is the backup target.** |
| `memories/`, `sessions/`, `plans/` | Agent's knowledge + history |
| `skills/`, `platforms/` | Installed skills + messaging integration configs |
| `SOUL.md`, `channel_directory.json`, `gateway_state.json` | Runtime + onboarding state |

## Install trace (reproducibility)

```bash
# 1. Dataset + .env seed on NAS (skip if restoring from backup)
ssh truenas_admin@192.168.1.65 'mkdir -p /mnt/fast/databases/hermes-data'
ssh truenas_admin@192.168.1.65 'cat > /mnt/fast/databases/hermes-data/.env <<EOF
OPENAI_BASE_URL=http://192.168.1.65:4000/v1
OPENAI_API_KEY=<LITELLM_MASTER_KEY from password manager>
# TELEGRAM_BOT_TOKEN=...   (added after @BotFather step; see Phase 3 Task 5)
# TELEGRAM_ALLOWED_USER_IDS=...
EOF
chmod 600 /mnt/fast/databases/hermes-data/.env'

# 2. Install the Custom App. Call times out client-side; job completes
#    in TrueNAS. Verify via `midclt call app.query`.
ssh truenas_admin@192.168.1.65 "midclt -t 300 call app.custom.create '$(python3 <<'PY'
import json
compose = '''services:
  hermes:
    image: nousresearch/hermes-agent:latest
    restart: unless-stopped
    command: [\"gateway\", \"run\"]
    env_file:
      - /mnt/fast/databases/hermes-data/.env
    volumes:
      - type: bind
        source: /mnt/fast/databases/hermes-data
        target: /opt/data
    ports:
      - target: 8642
        published: 8642
        protocol: tcp
        mode: host
'''
print(json.dumps({'app_name': 'hermes', 'custom_compose_config_string': compose}))
PY
)'"
```

### First-run setup wizard (required — DO NOT skip)

Hermes's `gateway run` command won't serve traffic until `hermes setup`
has run once. Setup is interactive — it prompts for API keys, model
choices, messaging platforms. **Must be done through the TrueNAS web shell
(stdin isn't available via `midclt`).**

Steps:
1. TrueNAS UI → **Apps** → click `hermes` card → scroll to **Workloads**
   section → click the **Shell** / terminal icon on the `hermes` container.
2. In the shell prompt, run:
   ```
   hermes setup
   ```
3. Walk through the wizard. The LLM provider step should point at the
   LiteLLM gateway (base URL + API key are already in `.env`, but confirm
   them in the wizard when asked).
4. Skip Telegram setup at this stage — we'll add it explicitly in a
   follow-up (Phase 3 Task 5) so the bot-token + allowlist convention is
   consistent.
5. Exit the shell. Hermes's gateway should now respond on `:8642`.
6. Verify: `curl -sS http://192.168.1.65:8642/health` from a LAN host
   should return `200` with a JSON body.

## Update / restart / remove

| Action | Command |
|---|---|
| Restart | `ssh truenas_admin@192.168.1.65 'midclt call app.redeploy hermes'` |
| Stop / start | `midclt call app.stop hermes` / `midclt call app.start hermes` |
| Remove (destructive — keep data) | `midclt call app.delete hermes '{"remove_images":false,"remove_ix_volumes":false}'` |
| Remove fully (wipes memory) | Delete as above **plus** `rm -rf /mnt/fast/databases/hermes-data/` — don't do this without a recent snapshot |
| Bump image tag | TrueNAS UI → Apps → hermes → Edit → update `image:` in the compose YAML → Save |

## Backup (Phase 8 scope)

The `/mnt/fast/databases/hermes-data/` dir is the entire stateful
surface. A snapshot task on the enclosing dataset covers it. Because the
state DB is live SQLite, a `.backup` via `sqlite3 state.db '.backup ...'`
is the canonical consistency-guaranteed logical copy; pair with the ZFS
snapshot for full recovery. See `nas/snapshots.md` — Phase 8 will wire
a nightly `.backup` to `bulk/backups/hermes/`.
